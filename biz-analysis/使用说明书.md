# biz-analysis Skill 使用说明书

> B端业务系统分析与设计 AI — 你的虚拟业务分析师

---

## 一、这个 Skill 能做什么

biz-analysis 是一个 Claude Code Skill，扮演**企业级核心业务分析师**角色。你给它原始业务资料（文档、聊天记录、截图、表格等），它会自动完成：

1. **理解你的业务** — 提取背景、目标、痛点、角色与 **DDD 限界上下文 (Bounded Context)**
2. **建立高清晰业务模型** — 抽取实体、分类聚合根与值对象，并**生成高质感 Mermaid ERD 关系图、状态流转图、时序协作泳道图**
3. **拆解功能清单** — 模块→页面→功能点→用户故事，并针对核心 P0 功能提供**高精度 Gherkin (Given-When-Then) BDD 验收用例**
4. **扫描深度风险** — 评估接口容灾、**敏感数据脱敏、RBAC/ABAC 安全选型、多租户 SaaS 隔离级别**与防重复提交幂等性
5. **辅助系统设计** — 建议聚合根表结构、**输出标准 OpenAPI Specification 3.0 API 协议 (YAML 格式)**、集成架构与微服务边界
6. **输出硬核分析报告** — Markdown 报告 + JSON 数据。在输出前，**自动运行本地原生 Node.js 校验脚本**进行 100% 数据一致性检查（可完美交接给 pm-html-pdt-fused 生成 PRD）

**它不是什么：**
- 不是 PRD 生成器（那是 pm-html-pdt-fused 的工作）
- 不是代码生成器
- 不是被动执行的文档工具 — 它会**主动提问、建议、反驳**

## 二、如何调用

### 方式 1：手动调用（推荐首次使用）

在 Claude Code 中输入：

```
/biz-analysis
```

然后描述你的业务需求。

### 方式 2：自动触发

直接在对话中描述业务需求，Claude 会自动识别并加载该 Skill。例如：

```
我们是一家制造企业，想做一个MES工单管理系统...
```

### 方式 3：带参数调用

```
/biz-analysis 我们公司需要一个CRM系统，管理客户和商机
```

## 三、使用前准备

### 最佳实践：准备好以下资料

| 资料类型 | 示例 | 为什么需要 |
|---|---|---|
| 业务背景文档 | 需求文档、招标书、项目方案 | 让 Skill 理解为什么要做 |
| 流程描述 | SOP 文档、会议纪要、微信聊天记录 | 让 Skill 理解业务怎么运转 |
| 角色说明 | 组织架构、权限说明 | 让 Skill 理解谁用什么功能 |
| 现有系统信息 | 老系统截图、数据库表结构、API 文档 | 让 Skill 理解要对接什么 |
| 业务规则 | 审批规则、计算公式、约束条件 | 让 Skill 理解业务约束 |

### 资料格式不限

支持：Markdown、Word、PDF、TXT、Excel、CSV、截图、原型图、聊天记录等。

**没有现成文档也没关系** — 直接在对话中描述你的需求，Skill 会主动提问补全缺失信息。

## 四、工作流程详解

Skill 会按以下 6 个阶段依次执行，每个阶段完成后**等你确认**才继续。

### Phase 1：资料接入与清洗

**Skill 做什么：**
- 读取你提供的所有资料
- 统一术语（如：客户=用户=商户 → 统一为「客户」）
- 提取结构化业务上下文

**你会看到：**
```
📋 业务上下文确认

我从你提供的资料中提取了以下信息：

业务背景：...
业务目标：...
当前痛点：...
已有系统：...
涉及角色：...
核心流程：...
业务规则：...

请确认以上信息是否正确。
```

**你需要做：** 确认信息是否正确，补充遗漏，修正错误。

> ⏸️ **HARD-GATE**：你确认后才会进入 Phase 2。

---

### Phase 2：业务理解与建模

**Skill 做什么：**
- 划分核心域、支撑域与通用域（DDD 战略设计）
- 抽取核心实体，区分聚合根（Entity）与值对象（Value Object）（DDD 战术设计）
- **自动绘制高清晰 Mermaid 关系图 (ERD)**，理清 1:N / N:M 结构
- **自动绘制 Mermaid 状态机图**，描述每个生命周期实体的流转规则与超时升级
- **自动绘制 Mermaid 时序泳道图**，清晰梳理跨部门/角色/系统的业务主流程
- 角色识别与安全权限分配 (RBAC/ABAC 架构设计)

**你会看到：**
- DDD 限界上下文定义表
- 实体与值对象战术划分表（含置信度标记 🟢🟡🔴）
- **Mermaid 动静态 ER 关系图**
- **Mermaid 状态流转图**
- **Mermaid 时序协作流程图**
- 角色权限配置矩阵表

**你需要做：** 确认实体是否完整、关系是否正确、限界上下文划分及流程是否有遗漏。

> ⏸️ **HARD-GATE**：你确认后才会进入 Phase 3。

---

### Phase 3：功能拆解与优先级

**Skill 做什么：**
- 将业务拆解为模块→页面→功能点
- 生成功能树
- 为每个功能写用户故事
- 按 MoSCoW 分优先级（P0-Must / P1-Should / P2-Could）
- 界定 MVP 范围（P0 = 最小可行产品）
- 分析功能依赖关系

**你会看到：**
- 功能清单表格（含模块、功能点、角色、优先级、用户故事）
- 功能树（文本格式）
- MVP 范围定义

**你需要做：** 确认功能是否完整、优先级是否合理、MVP 范围是否合适。

> ⏸️ **HARD-GATE**：你确认后才会进入 Phase 4。

---

### Phase 4：风险扫描与完整性检查（可跳过）

**Skill 做什么：**
- 从 8 个维度扫描风险（数据迁移、权限、流程闭环、接口依赖、上线切换、性能容量、审计合规、运维监控）
- AI 推导你没提但必须有的需求（如：草稿机制、防重复提交）
- 推导异常场景（如：审批人离职怎么办？接口失败怎么办？）
- 补全隐含业务规则
- 识别非功能性需求

**你会看到：**
- 风险评估表（高/中/低）
- 缺失需求列表（含 AI 推导理由和置信度）
- 异常场景清单
- NFR 清单

**你需要做：** 确认哪些风险要处理、哪些缺失需求要采纳。

> ⏸️ **HARD-GATE**：你确认后才会进入 Phase 5。

**跳过方式：** 对话中说「跳过风险扫描」或「直接进入系统设计」。

---

### Phase 5：系统设计辅助（可跳过）

**Skill 做什么：**
- 建议数据模型（基于聚合根事务边界设计主表、外键与值对象字段映射）
- 状态机战术落地设计
- **提供符合 OpenAPI Specification 3.0 标准的 YAML 接口协议**
- 集成架构与外部接口容灾设计（同步/异步、重试补偿机制、熔断降级方案）
- 建议微服务边界（基于 DDD 限界上下文划分服务边界）

**你会看到：**
- 数据模型表
- **OpenAPI 3.0 标准 API 协议 YAML 代码块**
- 集成架构与防灾防抖设计表
- 微服务边界清单

**你需要做：** 确认技术方案方向与 API 边界是否可接受。

> ⏸️ **HARD-GATE**：你确认后才会进入 Phase 6。

**跳过方式：** 对话中说「跳过系统设计」或「直接生成报告」。

---

### Phase 6：生成分析报告与自动化校验

**Skill 做什么：**
- 汇总前 5 个阶段的全部产出
- **自动调用本地验证脚本** `node scripts/validate_schema.js` 对 `analysis-data.json` 进行 100% 硬编码强校验，彻底拦截字段、枚举与引用的数据损坏与模型幻觉
- 生成完成度评分（满分100）
- 输出 3 个文件：
  - `analysis-report.md` — 人类可读的完整分析报告（内嵌高质感 Mermaid 渲染关系图）
  - `analysis-data.json` — 机器可读的结构化数据（包含 DDD 限界上下文、OpenAPI Spec 与 Gherkin BDD 验收用例）
  - `feature-tree.txt` — 纯文本功能树

**你会看到：**
- 自动化 Schema 校验成功提示 `🟢 Schema validation successful!`
- 完成度评分表及四维得分说明
- 交接下游提示

> ⏸️ **HARD-GATE**：你审阅并满意最终报告后，分析才算完成。

## 五、交互方式

### 阶段确认

每个 Phase 结束后，Skill 会停下来等你确认。你可以说：

- **确认**：「没问题，继续」/「确认」/「OK」
- **修改**：「XX 模块需要增加一个功能」/「优先级需要调整」
- **补充**：「还有一个角色叫 XX，需要加上」

### 中途修正

分析过程中你随时可以打断：

- **修正信息**：「等一下，审批不是2级，是3级」
- **新增需求**：「还需要一个知识库模块」
- **改变方向**：「不想做工单系统了，做 CRM」

Skill 会判断影响范围，可能需要回溯修正前面的阶段。

### 主动提问

Skill 会在分析过程中主动提问，例如：

- 发现矛盾信息时会问你哪个是对的
- 遇到模糊描述会追问细节
- 推导出缺失需求会问你要不要采纳

**回答技巧：**
- 明确回答即可，不需要长篇大论
- 不确定时可以说「你决定」或「按常见做法」，Skill 会自动推导
- 任何时候都可以说「这个先跳过，标记为待确认」

## 六、置信度标记

Skill 的每个分析结论都会标注置信度：

| 标记 | 含义 | 你需要做什么 |
|---|---|---|
| 🟢 高 | 基于明确信息推断，很确定 | 一般不需要核实 |
| 🟡 中 | 有依据但不100%确定 | 建议确认一下 |
| 🔴 低 | 基于行业类比推测 | 仅供参考，需要你判断 |

**示例：**
```
### 行业识别
🟢 高置信度：SaaS服务业
  依据：需求中明确提到「SaaS企业」

### 建议功能：知识库
🔴 低置信度：建议增加知识库模块
  依据：行业经验，需求中未提及
```

## 七、修改与迭代

分析完成后，你可以随时要求修改：

### 增量修改（不回溯）

```
「在功能清单里加一个数据导出功能」
「把SLA管理的优先级从P1提到P0」
```

Skill 只修改受影响的部分，不会重跑整个流程。

### 纠错修改（回溯重跑）

```
「客户不是工单的提交方，客户是工单的处理对象」
「行业识别错了，我们不是SaaS，是制造业」
```

Skill 会回溯到出错的阶段，修正后重跑后续所有阶段。

## 八、输出文件说明

分析完成后，在项目目录下生成 3 个文件：

### analysis-report.md

人类可读的完整分析报告，包含：
1. 业务上下文
2. 业务模型（实体/关系/状态机/流程/角色）
3. 功能清单（含优先级和用户故事）
4. 风险评估
5. 系统设计建议
6. 完成度评分

**适合：** 给团队评审、存档、作为 PRD 的输入。

### analysis-data.json

结构化 JSON 数据，供 pm-html-pdt-fused 消费。

**适合：** 程序化处理、交接给下游工具。

### feature-tree.txt

纯文本功能树，方便快速浏览。

**适合：** 快速分享、贴到会议纪要里。

## 九、与 pm-html-pdt-fused 的衔接

biz-analysis 和 pm-html-pdt-fused 是互补的：

```
biz-analysis（分析） → pm-html-pdt-fused（生成）
     ↓                        ↓
  业务理解                  PRD 文档
  功能清单        →        高保真原型
  系统设计                  页面标注
```

**衔接方式：**

分析完成后，Skill 会提示：

```
分析完成。如需生成 PRD 和原型，可调用 /pm-html-pdt-fused 并引用 analysis-data.json
```

你只需：

```
/pm-html-pdt-fused 基于 analysis-data.json 生成PRD和原型
```

pm-html-pdt-fused 会读取 analysis-data.json，自动使用其中的功能清单、业务模型、数据模型等信息。

## 十一、常见问题

**Q：资料太少怎么办？**
A：Skill 会评估输入充分度，如果信息不足会主动提问引导你补全。你也可以直接说「按行业常见做法处理」，Skill 会基于行业经验推导。

**Q：分析结果不对怎么办？**
A：随时可以打断修正。告诉 Skill 哪里错了，它会回溯修正受影响的阶段。

**Q：可以跳过某些阶段吗？**
A：Phase 1-3 不可跳过（核心分析），Phase 4-5 可跳过。说「跳过风险扫描」即可。

**Q：分析到一半关闭了怎么办？**
A：重新打开 Claude Code，描述之前的项目背景，Skill 会重新分析。建议在分析过程中让 Skill 定期保存进度到文件。

**Q：支持多系统同时分析吗？**
A：建议每次分析一个系统。如果需要分析多个相关系统，可以分别分析后在 Phase 5 中讨论系统间集成。

**Q：分析需要多长时间？**
A：取决于输入复杂度。简单系统（<20功能点）约 10-15 分钟，中等系统（20-100功能点）约 20-30 分钟，复杂系统（>100功能点）可能需要更长时间。

---

## 十二、最佳实践

1. **资料越丰富，分析越准确** — 尽量提供多角度的输入
2. **确认阶段认真看** — 每个 HARD-GATE 都是纠偏机会，别急着跳过
3. **善用置信度** — 🟢 的结论可以快速通过，🟡🔴 的需要你重点判断
4. **及时修正** — 发现错误立即说，不要等到最后
5. **先分析再生成** — 用 biz-analysis 做分析，再用 pm-html-pdt-fused 生成文档，质量更高
6. **保留 analysis-data.json** — 这是结构化数据，后续修改和交接都靠它
7. **关注进度提示** — 每个 Phase 开始时会显示当前进度和预估剩余时间
8. **HARD-GATE 死锁保护** — 如果连续 3 轮未明确确认，Skill 会给出二选一选项，避免无限等待
